## 1. 项目概述

### 1.1 一句话描述

为想建立微习惯的人提供极简追踪工具，通过“今日混合清单 + 稳定排序 + 口径透明统计 + 拼豆激励”提高坚持度。

### 1.2 目标用户

- 想用“极简但可追踪”的方式建立微习惯的人（学习/健康/生活）。
- 需要在“今日”同时管理每日固定习惯与当天临时事项的人。
- 对列表顺序与组织有要求的人：希望顺序稳定、可拖拽，并能用多标签快速聚合查看。

### 1.3 核心价值主张（P0）

- 执行极简：打开就能看到今天要做什么，并快速完成打卡/临时事项。
- 顺序可控：手动拖拽排序稳定可靠；规则排序不破坏手动顺序。
- 规则透明：统计/图表/关键名词口径在产品内可查（集中在“产品说明”）。
- 激励可感知：坚持会带来可视化进展变化，并可沉淀为快照回看。

### 1.4 产品原则

- 极简优先：主流程少打扰、少层级、少配置。
- 本地优先：默认离线可用与隐私友好（P0 不承诺多端同步）。
- 可解释优先：所有“为什么这样算/这样展示”都有对用户友好的解释入口。

## 2. User Stories & Scenarios（用户故事）

### 2.1 用户画像（P0）

- 画像A：轻量自律型。目标是做得久，不追求复杂计划；需要“打开就做”。
- 画像B：混合待办型。每天固定习惯 + 临时事项同时存在；希望统一清单管理。
- 画像C：秩序掌控型。对顺序、分类、回顾有要求；讨厌“排序乱跳/口径说不清”。

### 2.2 核心场景（关键路径）

- 场景S1：创建习惯 → 今日出现 → 立刻打卡一次。
- 场景S2：今日执行 → 多次目标逐步完成 → 记录备注（可选）→ 当日完成。
- 场景S3：补打卡 → 只对过去日期且该日应出现的习惯允许计入 → 明确提示限制原因。
- 场景S4：用标签组织 → 标签页“今日待办置顶”快速聚合 → 不受日历选中日期影响。
- 场景S5：回顾复盘 → 统计页切换周期 → 看趋势/榜单 → 在产品说明理解口径。
- 场景S6：激励沉淀 → 激励随坚持变化 → 关键节点保存快照 → 回看与管理。

## 3. Functional Requirements（功能需求）

### FR-1 习惯管理

- 支持创建/编辑：名称、图标、标签（可多选）、频率、每日目标次数、开始日期、备注（可选）。
- 支持状态：启用/归档/完成/删除；部分状态需提供明确的状态解释与可恢复策略。
- **归档语义**：
  - 归档操作当日：习惯仍在今日清单显示，标记为"已归档"状态，打卡按钮禁用
  - 归档次日及之后：习惯从今日清单隐藏
  - 恢复机制：仅能通过 设置/归档页 手动"启用"恢复
- 支持批量操作：批量启用/归档/删除。
- 支持搜索：按名称、标签、备注搜索习惯与临时事项。
- 搜索结果统计：显示搜索结果数量统计，帮助用户快速了解搜索范围。
- 删除语义：删除后不在当前列表出现，删除即彻底删除。
- 长按操作：支持长按习惯/事项触发操作菜单，包含编辑、归档、删除选项，删除和归档操作需弹出确认弹窗。

### FR-2 今日清单（按日期）

- 今日清单按"所选日期"展示内容：该日应出现的固定习惯 + 该日创建的临时事项 + 该日操作归档但尚未跨日的习惯（以归档状态展示，打卡按钮禁用）。
- 固定习惯与临时事项同列展示，支持混排与手动拖拽排序。临时事项与固定习惯的排序权重相同，均可自由拖拽调整顺序。
- 支持按日期查看历史清单（用于回顾与补打卡入口）。

### FR-3 打卡与补打卡

- 当日打卡：支持多次目标逐步完成展示；支持备注（可选）
- 打卡按钮状态逻辑：根据习惯的目标次数和当前完成状态，显示不同的按钮类型：
  - +1 按钮：目标次数大于1且已开始打卡的习惯，点击后增加已完成次数并更新进度显示
  - 打卡按钮：目标次数等于1的习惯，或目标次数大于1但未开始打卡的习惯，点击后标记习惯为已完成状态
  - 完成按钮：已完成当日目标的习惯，按钮变为灰色不可点击
- 补打卡：仅允许补到过去日期，不允许补未来。
- 补打卡限制：仅对“该日应出现”的习惯允许补打卡；不满足时需明确提示并阻止计入统计与分数。

### FR-4 频率与"应出现"（产品口径）

- 频率类型：每日 / 每周（按星期）/ 每月（按日期）/ 自定义（每 N 天）。
- 时间口径：以设备本地日历天为边界（按用户对"今天/昨天"的直觉）。
- **自定义频率判定公式**：当日期 >= 开始日期时，(date - startDate) 的天数差 % N == 0 则应出现（详见 TDD 3.2）。
- 规则变更影响：变更后对"哪些天应出现"的展示与统计需一致且可解释；历史记录保留供回顾，但不符合最新规则的历史打卡不计入当前口径。

### FR-5 临时事项（One-off）

- 支持基础信息（名称/图标/标签）与目标次数、逐步完成、备注（可选）。
- 默认插入当日列表末尾；允许与固定习惯混排并拖拽。
- 视觉标识：临时事项在列表中以虚线边框区分显示。
- 完成操作：点击完成按钮标记临时事项为已完成状态。

### FR-6 标签系统与标签页聚合

- 标签支持多选绑定到习惯与临时事项。
- 标签页为“双区结构”：置顶区（今日待办）+ 其余区（不应出现）。
- 标签页的“今日”严格指本地今天，与用户在日历中选中的日期无关。
- 标签页“其他”区域的习惯（不应出现的习惯）也可以打卡并计入权重。
- 标签管理功能：支持添加新标签，标签名称长度限制为10个字符。
- 标签编辑/删除：支持长按标签触发操作菜单，包含编辑、删除选项，删除操作需弹出确认弹窗。

### FR-7 排序系统（手动 + 规则）

- 手动排序：拖拽可用且在重启后保持稳定。临时事项与固定习惯一样参与手动排序，排序权重相同。
- 规则排序：提供排序选项弹窗，包含手动排序、按名称排序、按创建时间排序、按完成度排序等常用规则排序入口，仅影响当前展示，不覆盖手动排序。
- 作用域隔离：不同页面/不同日期/不同标签页的排序互相独立，不要求跨视图一致。

### FR-8 日历交互

- 支持月/年视图切换与选日。
- 日期格提供任务简单的圆点标记展示（最多6个圆点，未完成打卡的圆点在前）。
- **圆点标记统计规则**：仅统计该日"应出现且未归档"的习惯数量，已归档习惯不计入。
- 选日后能快速查看该日清单，并可进行补打卡。
- **日历页下方清单**：显示该日应出现的习惯 + 临时事项 + 该日归档的习惯（以归档状态展示，打卡按钮禁用）。

### FR-9 统计系统

- 支持日/周/月/年维度切换，通过周期选择器快速切换视图。
- 提供多维度统计视图：
  - 日视图：展示今日进度、完成总数、打卡总次数、未打卡事件及当日打卡列表。
  - 周视图：展示本周统计指标、完成趋势图（柱状图）及习惯榜单（支持按次数/完成率/共打排序）。
  - 月视图：展示本月统计指标、习惯榜单及月度打卡日历。
  - 年视图：展示本年统计指标、习惯榜单及年度打卡日历。
- 支持日期选择功能，可选择不同日期范围查看历史统计数据。
- 提供趋势与榜单类视图；统计口径与解释需可达（设置页产品说明入口）。
- 与底部导航栏集成，点击统计图标可快速进入统计页。

### FR-10 激励（拼豆）

- 激励页展示当前进度、阶段与解锁状态（阈值与细节由产品说明承载并可迭代）。
- 激励页面结构：
  - 标签1「当前图片」：展示当前正在解锁的图片及阶段进度
  - 标签2「全部图案」：展示系统所有图案，包括已解锁、已完成、正在进行的图案，支持类型筛选
  - 标签3「照片箱子」：展示已保存的快照列表，支持搜索和管理
- 权重获取：任意习惯或临时事项在某日完成当日目标时获得 1 权重；每个项目每天最多 1 权重；权重一旦获得不回退。

#### 快照（照片箱子）功能

**功能定位**：

- 快照记录的是"当前解锁图片的阶段成果"，与具体习惯无关
- 全局权重推进图片阶段，阶段4和阶段5自动触发保存提示

**保存时机**：

- **自动提示**：
  - **阶段4快照**：当前图片权重达到12（图1完全显现）时，提示保存
  - **阶段5快照**：当前图片权重达到15（图2进化完成）时，提示保存
- **手动保存**：任何阶段（0-5）用户都可手动点击"保存当前进度"按钮保存快照
- **限时保存**：每个阶段都是限时的，该阶段的画面仅在当前阶段可保存，进入下一阶段后不可回溯保存上一阶段的画面
- **限时提示机制**：每个阶段都显示提示提醒用户及时保存，传达"当下即永恒"的理念，鼓励用户珍惜每一个进步时刻

**存储限制**：

- 单用户最多保存50张快照（LRU策略，超限删除最旧的阶段4快照，保留阶段5）
- 存储空间显示：实时显示已用/上限数量（如"7 / 50"），帮助用户了解存储使用情况

**查看方式**：

- 列表视图：缩略图 + 图片名称 + 阶段标记（阶段4/阶段5）+ 保存日期
- 大图模式：点击缩略图全屏查看，支持双指缩放查看拼豆细节
- 导航功能：大图模式下支持上一张/下一张切换浏览
- 搜索功能：支持按快照名称、保存日期、阶段关键词搜索过滤

**删除策略**：

- 单条删除，删除后不可恢复
- 删除快照不影响已获得的全局权重，不影响当前图片阶段

**与习惯删除的关系**：

- 删除习惯仅减少未来获得权重的来源，不影响已保存的快照
- 快照不关联任何习惯，仅关联图片ID（如"橘子象-1号"）

* 删除习惯/临时事项后，已获得的全局权重永久保留（不回退），当前图片的阶段进度不受影响。已保存的快照（阶段4/阶段5）仍可回看，快照展示图片名称而非习惯名称。

#### 拼豆激励详细规则（P0）

- **阶段定义（0-5）**：
  - 阶段0：权重0，空白1：1白色底板
  - 阶段1：累计权重1-3
  - 阶段2：累计权重4-6
  - 阶段3：累计权重7-9
  - 阶段4：累计权重10-12
  - 阶段5：累计权重13-15
- **显现逻辑**：
  - 阶段0-4：从白板逐渐显示无\_1后缀的基础图片（如1.jpg、2.jpg等）
  - 阶段4-5：从基础图片逐渐过渡到对应的\_1后缀版本（如从1.jpg过渡到1\_1.jpg）
- **图片解锁机制**：
  - 用户在激励页面选择要解锁的图片
  - 同类型内，前一张图片达到阶段5（即已完成状态）后自动解锁下一张
  - 类型定义：
    - 类型按图片编号分组，如第一类、第二类等
  - 每个类型包含多个基础图片（如1、2、3等）及其对应的高清版本（如1\_1、2\_1、3\_1等）
- **类型解锁机制**：
  - **解锁条件（需同时满足）**：
    1. 前一类型完成至少2张图片（达到阶段5）
    2. 达到该类型的累计总权重阈值
  - **各类型解锁条件**：
    | 类型       | 解锁条件                            | 累计总权重阈值  |
    | -------- | ------------------------------- | -------- |
    | 类型1      | 默认解锁                            | -        |
    | 类型2      | 类型1完成≥2张图片 且 累计总权重≥15           | 15       |
    | 类型3      | 类型2完成≥2张图片 且 累计总权重≥30           | 30       |
    | 类型4      | 类型3完成≥2张图片 且 累计总权重≥45           | 45       |
    | 类型5      | 类型4完成≥2张图片 且 累计总权重≥60           | 60       |
    | 类型N（N≥2） | 类型(N-1)完成≥2张图片 且 累计总权重≥15×(N-1) | 15×(N-1) |
  - **通用公式**：对于第N个类型（N≥2），解锁需要满足：
    - 类型(N-1)已完成图片数 ≥ 2
    - 累计总权重 ≥ 15 × (N-1)
- **权重冻结**：基于打卡当日目标判定，一旦获得永不回退，防止降低目标后倒灌权重
- **数据模型**：currentPictureId, currentPictureWeight(0-15), globalTotalWeight, unlockedTypes, unlockedPictures, completedPictures, currentType, currentPictureNumber, completedCountInType

### FR-11 提醒与免打扰

- 每个习惯支持提醒开关与最多 3 个提醒时间点。
- 点击提醒进入应用并定位到今日清单。
- 提醒豁免：当日已完成或习惯处于归档/完成状态时不提醒。
- 全局免打扰：免打扰时段内的提醒将顺延到免打扰结束后立即发送。
- **Capacitor本地通知实现**：iOS需配置NSUserNotificationUsageDescription，Android 12+需SCHEDULE\_EXACT\_ALARM权限
- **提醒内容格式**：包含习惯名称与今日完成进度（如"喝水 - 已完成2/8杯"）
- **提醒豁免规则**：
  - 已完成习惯（dayCompletion=1）不提醒
  - 豁免判定与weightEarned无关

### FR-12 设置与产品说明

- 设置页需提供"产品说明"入口，集中解释关键名词与口径，避免主流程信息噪音。
- 产品说明建议目录：
  - 统计与图表口径说明
  - 频率与应出现规则
  - 归档/删除等状态语义
  - 排序机制（手动/规则共存）
  - 标签聚合规则
  - 激励规则
- 已归档的习惯，其历史数据仍可查看；归档当日仍显示在今日清单（标记为归档状态），次日起隐藏；恢复需通过 设置/归档页 手动"启用"。

#### FR-12.1 外观设置

- **深色模式**：支持跟随系统/浅色/深色三种模式切换
  - 默认：跟随系统设置
  - 切换后实时生效，无需重启应用
  - 所有界面元素需适配深色模式显示
  - 深色模式颜色方案见设计系统文档
- **主题颜色**：支持10种预设主题颜色选择
  - 默认：温暖橙(#D97757)
  - 可选颜色：
    - 蓝色(#6A9BCC)
    - 绿色(#788C5D)
    - 紫色(#9B8AA3)
    - 粉色(#C49BA8)
    - 黄色(#C9B896)
    - 青色(#7DA3A8)
    - 橙色(#E67E22)
    - 深紫(#8E44AD)
    - 深绿(#27AE60)
  - 主题颜色影响范围：
    - 主按钮背景色
    - 开关选中状态
    - 底部导航选中图标
    - 强调文字颜色
    - 选中状态指示器
- **实时预览**：选择主题颜色时，预览区域实时展示效果
  - 预览内容包括：按钮、图标背景、开关、文字颜色
  - 预览区域展示典型UI组件组合
  - 切换颜色时无需保存即可看到效果

### FR-13 数据导出与备份（P1）

- **导出格式**：
  - JSON格式（默认）：包含完整数据结构，文件名格式 `habit_backup_YYYYMMDD_HHMMSS.json`
  - CSV格式（可选）：仅统计数据，文件名格式 `habit_stats_YYYYMMDD.csv`
- **导出范围**：
  - 完整备份：所有习惯+历史打卡+标签配置+设置项
  - 单习惯导出：仅选中习惯的历史记录
- **脱敏导出选项**：可移除提醒时间、打卡备注等敏感信息
- **恢复时的文件验证**：自动检测导出类型（full/sanitized），脱敏文件恢复时需提示警告
- **数据备份与恢复（P1）**：支持从JSON备份文件进行"覆盖恢复"，导入前可预览摘要并二次确认

### FR-14 跨平台与技术方案

- **主分发渠道**：Web应用作为默认技术方案，支持iOS/Android/桌面浏览器，通过浏览器直接访问使用，无需应用商店审核
- **功能增强Capacitor**：作为原生能力补丁，提供本地通知等系统级功能支持
- **响应式自适应布局**：
  - **未来扩展考虑**：PWA功能（Service Worker离线缓存、manifest配置等）可作为后续迭代方向，当前阶段优先保证Web端核心体验
  - **当前实现**：基于Vue 3 + Vite的纯Web应用架构，数据本地存储于IndexedDB
  - 采用响应式自适应布局，不使用手机设备框架模拟
  - 原型中的手机框架仅用于预览移动端显示效果，不是设备模拟器
  - 移动端（< 768px）：单栏垂直布局，内容宽度100%，底部Tab导航
  - 桌面端（≥ 768px）：同样是垂直布局，内容区随屏幕宽度扩展，最大宽度1200px居中
  - 所有功能在任意宽度下均完整可用，无功能差异
- **移动端基准设计**：以 iPhone 14/15（393px）和 Android 中屏（360px）为主要设计基准
- **桌面端适配**：大屏显示时内容区最大宽度1200px居中，垂直布局随视口扩展
- **安全区域**：适配 iPhone 刘海屏（safe-area-inset-top）和 Android 导航栏
- **性能优化**：懒加载习惯图标/拼豆图案；资源压缩；文件箱子快照 LRU 缓存（最多 10 个在内存）
- **前端技术栈**：HTML5 + CSS3 + Tailwind CSS + Vue 3 + TypeScript + Vite
- **响应式实现**：
  - 使用 Tailwind CSS 响应式断点（sm:640px, md:768px, lg:1024px）实现布局自适应
  - 主题颜色通过 CSS 变量动态注入，支持运行时切换
  - 用户偏好存储于 IndexedDB，应用启动时读取并应用
  - 监听系统主题变化（prefers-color-scheme）实现跟随系统模式
  - **PWA扩展预留**：如需后续支持PWA，可集成vite-plugin-pwa插件，配置Service Worker和manifest.json
- **新建功能**：提供新建选项弹窗，支持新建习惯和新建临时事项

## 4. UI/UX Requirements（界面与交互）

本章提供通用交互规范；**高保真可交互原型已完成并验收**，详见第 2.3 节「用户旅程与原型交付物」。

### 4.0 原型交付状态

所有关键页面原型（今日清单 5 状态、标签页、习惯编辑、日历、统计、激励、设置）已完成 HTML 高保真实现，共计 17 个交互文件。详细清单与验收标准见第 2.3 节。

### 4.1 全局交互规则

- 主要流程的关键动作不超过 2 次点击可达（例如：今日打卡、添加临时事项、创建习惯）。
- 触控目标足够大（适配移动端），重要操作避免靠近系统手势边缘。
- 所有"限制/阻止"场景给出明确原因与可执行下一步（例如：补打卡受限、权限被拒）。
- **响应式交互一致性**：
  - 移动端（< 768px）和桌面端（≥ 768px）使用相同的响应式组件，功能完全一致
  - 布局随视口宽度自适应变化，单栏垂直布局在大屏上最大宽度1200px居中
  - 原型中的手机框架仅用于预览移动端显示效果，不是设备模拟
  - 拖拽排序在触屏设备和鼠标设备上均可正常工作

### 4.2 列表与拖拽排序

- 列表项信息层级清晰（图标/颜色/名称/状态/进度）。
- 拖拽排序有明确把手或可拖拽提示；拖拽过程中有占位反馈；完成后有轻反馈。
- 规则排序开启时有明显提示，并提供一键返回“自定义（手动）”排序入口。

### 4.3 关键界面状态（必须覆盖）

- 空状态：无习惯、无标签、无统计数据、照片箱子为空等，并提供引导（创建/添加/查看示例）。
- 加载态：首次进入、切换日期、切换统计维度等。
- 错误态：保存失败、提醒权限未开启等，提供可执行修复路径。
- 权限态：提醒权限首次请求、拒绝后的再引导与入口。
- 动态时间显示：实时更新当前时间和日期，提升用户体验。
- 空状态引导：无数据时显示引导按钮，引导用户创建第一个习惯。
- 主题切换状态：
  - 深色模式切换时，界面元素平滑过渡，无闪烁
  - 主题颜色切换时，预览区域实时更新，保存后全应用生效
  - 系统主题变化时，若用户选择"跟随系统"，应用自动适配

### 4.4 文案与新手引导

- 关键术语统一：今日/应出现/补打卡/归档/完成/删除。
- 首次使用引导：最短路径完成“创建第一个习惯并打卡一次”。
- 产品说明入口可发现；说明文本简短、可检索。

### 4.5 可访问性

- 对比度满足 WCAG 2.1 AA；颜色不作为唯一信息载体。
- 支持系统级减少动态效果；关键反馈在无动效时仍可感知。
- Web场景支持基础键盘导航与屏幕阅读器标签。
- **PWA可访问性（未来扩展）**：如需支持PWA，需确保离线状态下辅助功能正常工作，Service Worker更新时保持可访问性状态

## 5. Non-Functional Requirements（非功能需求）

### 5.1 性能

- 冷启动 < 2s（P0 基线）。
- 关键页面切换响应 < 0.5s（P0 基线）。
- 100+ 习惯/大量历史数据下依然可用且无明显卡顿（P0 基线）。

### 5.2 兼容性

- iOS 14+ / Android 10+ / Chrome 90+（P0 基线）。

### 5.3 可访问性

- 目标满足 WCAG 2.1 AA（对比度、可读性、减少动效等）。

### 5.4 隐私与数据

- 数据默认仅本地存储，不上传。
- 不在可见日志/提示中泄露用户敏感内容（如完整备份文本）。

### 5.5 安全需求

- 所有用户个人信息，比如用户名、性别、出生日期等内容，需要在接口层加密传输。
- 所有前端页面，需要使用 HTTPS 协议。

### 5.6 可用性需求

- 操作系统支持：当下所有主流机型；
- 信息支持缓存机制，只要加载完成，网络中断后仍可显示信息；
- 人性化帮助：在重要的功能入口处提供相应的帮助/反馈入口；
- 用户可控：页面出口应明显、退出时需用户手动确认，防止用户误操作；

### 5.7 产品升级维护需求

- 根据产品初期特色梳理产品基本需求，根据产品发展期收集整理用户反馈和市场新兴可兼容技术与功能热点等新需求，根据产品高速增长期功能与技术点转型和收益业务为主要战略发展方向增添企业需求
- 基本管理办法通过需求评审会将以提出需求分为四类，近期迭代需求、后期迭代需求、未来需求和淘汰需求
- 召开需求评审会配合产品的健康迭代，“31”（周）迭代小功能，“21”（月）迭代大版本需求，提前整理需求，与技术团队、公司管理层召开需求评审会，过需求
- 需求评审会召开时间，评审时间为当前小版本迭代期最后一周（测试阶段），大版本迭代完成后第三周（修复bug阶段）例如：1.1.2版本预计发布日期前5个工作日召开。

## 6. Open Questions & Out of Scope（边界与排除项）

### 6.1 明确排除（P0）

- 多设备同步与账号体系（P1 以后再评估）。
- 社交分享、社区、好友互动、公开链接/二维码分享。
- 番茄钟/倒计时等计时类功能。
- 第三方集成与深度智能建议/预测分析。

### 6.2 待决策（P1）

- 备份/恢复策略：仅"覆盖恢复"还是需要"合并恢复"。

### 6.3 已知风险（P0）

- 频率变更导致用户困惑：需要明确提示“按新规则重算”，并在回顾中做可理解标识。
- 过度打扰：提醒默认克制，提供免打扰与完成豁免，并监控关闭提醒行为。

## 7. Acceptance Criteria（验收标准）

- AC-1：用户创建每日习惯后，该习惯在今日清单出现且可打卡。
- AC-2：用户完成当日目标后，习惯显示为完成状态；当日不再触发该习惯提醒。
- AC-3：用户拖拽排序后，重启应用该视图顺序保持不变。
- AC-4：用户添加临时事项后，该事项仅出现在对应日期的今日清单，并可打卡与混排排序。
- AC-5：用户在标签页看到“双区结构”；“今日待办”的“今日”不受日历选中日期影响。
- AC-6：用户仅能对过去日期补打卡，且仅应出现日可计入；非应出现日会被阻止并说明原因。
- AC-7：用户删除习惯后，该习惯不再出现在活跃列表。
- AC-8：用户为习惯设置提醒与免打扰后：到点可触发提醒；免打扰/完成状态按规则豁免生效；点击提醒可进入今日清单。
