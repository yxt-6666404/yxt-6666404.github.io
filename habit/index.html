需求发现与验证

触发点

2026年初Vibe Coding爆火，我有技术背景但做毕设时经常凌晨3-4点写代码，深刻体会到工具效率对执行力的影响。当时使用一款鸿蒙版微习惯App，遇到5个痛点：

  • 排序经常Bug，自定义排序没实现，还是按新增顺序排
  • 分数算法不透明，不知道今天为什么得这个分
  • 没有多标签分类，工作和健身习惯混在一起
  • 想要临时记个事，但App不支持临时事项
  • 进度展示全是数据太枯燥，没有成就感

用户验证与竞品分析

访谈5位朋友，4位有类似痛点

调研了4款主流产品（微习惯、微笑ToDo、Forest、Todoist），发现市场两极分化——要么太游戏化（Forest断签惩罚、积分排名造成焦虑），要么太工具化（冰冷数据列表，但分数不透明、缺多标签、无法临时记事项）。

竞品对比结论

微习惯

纯工具型，排序Bug，分数不透明，无标签，无临时事项

微笑ToDo

工具+情感化，但习惯/待办分离，无程度打卡，无多标签

Forest

专注场景强但习惯追踪弱，有惩罚机制增加焦虑

差异化定位：零压力伴侣型习惯追踪工具

渐成画迹 = 微习惯理念（小目标、无压力）+ 像素画渐进激励（从模糊到清晰）+ 零压力设计（权重冻结、无断签惩罚）

定位在"游戏化过头"和"工具感太强"之间，强调'自我对话'而非外部比较。满足"零压力+视觉激励+算法透明+多标签+临时事项"。

名字由来：「渐成画迹」

从Trae AI多轮对话后10个选项中挑选——渐进养成，留下痕迹。符合微习惯'小步快跑'的理念，'画迹'呼应拼豆像素画的视觉主题。每一次打卡都是一笔，日积月累，渐成画迹。

核心功能

1

今日混合清单

应出现算法（自动判断今天该做什么）+ 双区标签页（今日待办置顶区 / 其余区）+ 稳定排序（手动拖拽持久化 + 规则排序共存不覆盖）。固定习惯与临时事项混合同屏，临时事项用虚线边框区分，同等排序权重。统一 TodayListItem 数据结构，通过 kind 字段区分类型。排序状态按视图key隔离持久化（today_YYYY-MM-DD），退出后恢复手动顺序。

图1：今日清单与标签页面原型（上排：今日页面两种状态 / 下排：标签页面两种分类）

2

打卡与补打卡

核心公式 dayCompletion = min(1, doneCount / target)，目标≥2时逐步+1。补打卡"双限制"策略：仅过去日期可补 + 仅「应出现日」可补；非应出现日被拦截并弹窗说明原因（如"该习惯按每周一设置，此日期不应出现"）。复用 shouldAppear 算法确保补打卡限制与日常展示逻辑一致。打卡按钮三态视觉反馈（+1态 / 完成态 / 禁用态）。

图2：日历回顾页（月视图圆点标记 + 选中日详情清单 + 补打卡入口）

3

拼激励系统

核心差异化功能。每次打卡获得1权重，15权重分5阶段渐进解锁（阶段1:3、阶段2:6、阶段3:9、阶段4:12、阶段5:15）。clarity递进公式：0.4 + 0.6 × min(1, weight/15)。权重冻结机制：weightEarned 一旦为 true 永不回退，保护用户成就感（损失厌恶心理学）。三标签结构：当前阶段（主拼图+预览）+ 全部图案（已解锁/待解锁）+ 照片箱子（阶段4/5自动弹出快照保存，最多50张LRU）。CSS Grid实现225个DOM节点渐进显现（blur/opacity），非Canvas。

图3：拼豆激励页（上排：阶段4快照弹窗 / 照片箱子 / 下排：全部图案类型筛选）

4

标签系统

多标签绑定（habitTagIds: string[]），支持交叉分类。标签页双区结构：置顶区=应出现且未完成，其余区=不应出现或已完成；不受日历选中日期影响（标签页"今日"固定为本地今天）。标签长按管理（编辑/删除色）。排序系统：手动拖拽 / 名称A-Z / 创建时间 / 完成进度四种规则，手动排序优先级高于规则排序永不覆盖。用户反馈"置顶区就是我今天要做的，找习惯快多了"。

图4：标签页双区布局（左：健康分类筛选 / 右：全部标签+PRD说明弹窗）

5

统计系统

支持日/周/月/年四维周期切换。日视图展示今日进度环+打卡列表；周视图展示完成趋势柱状图+习惯榜单（次数/完成率/共打排序）；月视图展示月度打卡日历+习惯榜单；年视图展示年度打卡热力图+习惯榜单。所有统计口径透明可查（产品说明入口），数据本地存储于IndexedDB。

图5：统计页四维视图（上排：日视图 / 周视图 / 下排：月视图 / 年视图）

6

设置与数据管理

核心功能：产品说明（统计口径/频率规则/归档语义集中解释）+ 归档习惯库（管理已归档习惯，支持手动启用恢复）。提醒与通知：全局提醒开关 + 免打扰时段设置（22:00-08:00）。数据管理：导出/导入数据（JSON备份）+ 清空数据。支持深色模式切换与10种主题颜色自定义，实时预览生效。

图6：设置页（产品说明 / 归档习惯库 / 提醒通知 / 数据导出导入）

产品定义与技术架构

P0核心问题定义

① 怎么让用户愿意开始？ → 降低启动门槛（无需注册登录、3分钟创建习惯）

② 怎么让用户愿意持续？ → 正向激励而非负向惩罚（权重冻结、拼豆渐进）

③ 怎么让用户信任数据？ → 统计透明、口径清晰（产品说明集中解释）

Kano模型 + 2×2矩阵决策

P0

高价值低成本（拼豆激励/临时事项混排/标签系统）→ 差异化竞争；低价值低成本（打卡/提醒/日历）→ 产品底线必须做

P1

高价值高成本（云同步）→ Repository模式预留扩展，P1可低成本接入Supabase

排除

反向需求（社交/排行榜/番茄钟）→ 做了降低满意度，与"零压力自我对话"定位冲突

技术选型决策

Vue 3 + TypeScript + Pinia

Composition API适合复杂状态管理，TS类型安全减少AI代码bug

IndexedDB + Dexie.js

容量大（>5MB）、Promise封装代码量减少60%，接口存储无关可低成本迁移Supabase

Capacitor

无需审核快速迭代，Capacitor作为提醒功能补丁（原生通知）

CSS Grid像素渲染（非Canvas）

225个DOM节点完全可控，调试方便，Tailwind直接调blur/opacity实现渐进效果

核心算法

① shouldAppear 判定

today >= startDate && 频率匹配 && !paused && !archived

② dayCompletion 计算

dayCompletion = min(1, doneCount / target)

③ 权重冻结

canEarnWeight: if(alreadyEarned) return false; return doneCount >= target

Bug修复记录

发现2个关键Bug并修复：

Bug1：补打卡"应出现日"校验

错误：补打卡时未校验频率，导致非应出现日也能补打卡
修复：复用 shouldAppear 算法统一校验补打卡资格
影响：修复后补打卡限制逻辑100%通过AC-6验收

Bug2：拼豆页面状态同步

错误：打卡后权重增加，但clarity未同步更新
修复：Pinia Store触发视图刷新
影响：225个DOM节点blur/opacity即时响应
