### 3.1 像素激励系统

#### 1. 功能定位

**一句话**：通过"像素画渐进解锁"机制，让用户在坚持习惯的过程中获得可视化成就感，解决"断签打击导致放弃"的痛点。

#### 2. 设计决策

| 决策点      | 选项A        | 选项B        | 最终选择    | 决策理由                                        |
| -------- | ---------- | ---------- | ------- | ------------------------------------------- |
| **激励形式** | 积分排名（竞争式）  | 像素画解锁（收集式  | **选项B** | 竞品分析显示排名会让80%用户感到挫败；收集式更符合"自我提升"心理，无社交压力    |
| **权重规则** | 超额完成多给权重   | 每日每项目最多1权重 | **选项B** | 防止用户为了刷权重而过度打卡；保护健康习惯养成，避免 gamification 成瘾  |
| **断签处理** | 权重回退（惩罚机制） | 权重冻结（保护机制） | **选项B** | 基于损失厌恶心理学：人们更在意已得利益的损失；权重冻结保护用户成就感，降低断签后放弃率 |

#### 3. 技术实现

**关键技术方案**：

```typescript
// 核心算法：阶段计算（algorithms.ts）
export function calculateStage(weight: number): number {
  if (weight <= 0) return 0
  if (weight <= 3) return 1
  if (weight <= 6) return 2
  if (weight <= 9) return 3
  if (weight <= 12) return 4
  return 5
}

// 权重冻结策略：Checkin记录存储weightEarned布尔字段
// 一旦为true永久保留，与后续目标变更无关
// 具体实现涉及数据库查询逻辑，不在前端代码里
```

**为什么可行**：

- 权重计算基于打卡当日目标，存储在 Checkin 记录的 `weightEarned` 字段
- 阶段切换使用 CSS clip-path 实现像素画渐进显现，无需复杂动画
- 类型解锁采用"完成2张+权重阈值"双重条件，平衡进度与成就感

#### 4. 设计方案演进

| 阶段        | 方案                                                | 问题/反馈                                          | 演进           |
| --------- | ------------------------------------------------- | ---------------------------------------------- | ------------ |
| **第一版原型** | 用Canvas画像素画，15×15网格，每个像素独立控制                      | 调试时发现颜色调整困难、动画卡顿；朋友试用后说"看不清是什么图案"，因为Canvas像素太小 | 改用CSS Grid实现 |
| **第二版原型** | CSS Grid，225个DOM节点，每个节点独立控制opacity/blur           | 调试更方便，可以单独调整每个"像素"的模糊度                         | 增加5阶段渐进效果    |
| **最终版**   | 5阶段渐进显现（blur 10px→5px→2px→opacity 0.9→清晰）+ 权重冻结机制 | 测试用户反馈"像在开盲盒，想看到完整图片"                          | 增加照片箱子保存功能   |

### 3.2 今日清单（习惯+临时事项混排）

#### 1. 功能定位

**一句话**：让用户在"今日"视图中统一管理固定习惯和临时事项，解决"多个App切换"的碎片化痛点。

#### 2. 设计决策

| 决策点      | 选项A           | 选项B       | 最终选择    | 决策理由                                              |
| -------- | ------------- | --------- | ------- | ------------------------------------------------- |
| **事项类型** | 只有固定习惯        | 习惯+临时事项混排 | **选项B** | 用户调研显示：5位测试用户中4位（80%）表示每天既有固定习惯（阅读），也有临时事项（开会、买药） |
| **展示方式** | 分开展示（习惯区+事项区） | 统一列表混排    | **选项B** | 减少认知负担；临时事项用虚线边框区分，视觉层级清晰                         |
| **排序权重** | 固定习惯优先        | 同等权重可拖拽   | **选项B** | 用户希望对今日任务有完全掌控感；临时事项可能更紧急                         |

#### 3. 技术实现

**关键技术方案**：

```typescript
// 今日列表构建（todayList.ts）
export function buildTodayListItems(params: {
  date: string
  habits: Habit[]
  oneOffs: OneOffTask[]
  checkins: Checkin[]
  viewSortOrders: SortOrderItem[]
  sortMode: TodaySortMode
}): TodayListItem[] {
  // 1. 获取应出现的习惯
  const habitsForDate = habits.filter(h => shouldAppear(h, date))
  
  // 2. 获取当日临时事项
  const oneOffsForDate = oneOffs.filter(o => o.date === date)
  
  // 3. 合并为统一列表项
  const habitItems: TodayListItem[] = habitsForDate.map(h => ({
    kind: 'habit',
    id: h.id,
    name: h.name,
    // ...
  }))
  
  const oneOffItems: TodayListItem[] = oneOffsForDate.map(o => ({
    kind: 'oneOff',
    id: o.id,
    name: o.name,
    // ...
  }))
  
  // 4. 应用排序（手动/规则）
  return mergeAndSort([...habitItems, ...oneOffItems], viewSortOrders, sortMode)
}
```

**为什么可行**：

- 使用统一的数据结构 `TodayListItem`，通过 `kind` 字段区分类型
- 临时事项用虚线边框（`border-dashed`）视觉区分，不打断列表流
- 排序系统支持视图隔离（今日清单、标签页各自独立排序状态）

#### 4. 设计方案演进

| 阶段        | 方案                      | 问题/反馈                       | 演进            |
| --------- | ----------------------- | --------------------------- | ------------- |
| **第一版原型** | 临时事项和习惯分开显示，习惯在上，临时事项在下 | 用户反馈"要看两个地方，不方便"            | 合并为统一列表       |
| **第二版原型** | 统一列表，但临时事项默认排在最后        | 用户反馈"紧急的临时事项要手动拖到前面，麻烦"     | 临时事项与习惯同等排序权重 |
| **第三版原型** | 同等权重可拖拽，但排序后重启App顺序丢失   | 用户反馈"每次打开顺序都变，很烦"           | 引入排序持久化       |
| **最终版**   | 排序持久化 + 规则排序不覆盖手动排序     | 用户反馈"终于不用在习惯App和待办App之间切换了" | 定稿            |

#### 5. 验证结果

- **原型测试**：用户反馈概括："临时事项和习惯在一起方便多了，不用切换App"
- **AC通过情况**：AC-3（排序持久化）、AC-4（临时事项混排）已完成
- **代码覆盖**：todayList.ts 核心逻辑已通过单元测试

#### 6. 面试话术

**【可直接背诵】STAR故事版（100字）**：

> **Task**：设计一个统一的"今日"视图，让用户在一个页面完成所有当日任务管理。

> 2. **方案设计**：采用"混排+视觉区分"——习惯和临时事项在同一列表，临时事项用虚线边框标识
> 3. **排序设计**：两者同等排序权重，支持拖拽调整；规则排序（按名称/进度）仅影响当前展示，不覆盖手动排序

#### 7. 预埋追问点

1. **临时事项和习惯的数据模型有什么区别？** → 习惯有频率配置（每日/每周/自定义），临时事项只有日期；但共享基础字段（名称/图标/标签/目标次数）
2. **排序持久化怎么实现的？** → Settings store 存储 viewSortOrders，key 格式为 `today_YYYY-MM-DD` 或 `tag_<tagId>_todo`
3. **规则排序和手动排序怎么共存？** → 规则排序仅改变当前展示顺序，退出后恢复手动顺序；手动排序变更时持久化到存储
4. **临时事项过期后怎么处理？** → 从对应日期的 viewSortOrders 中移除，但数据保留供历史回顾

#### 8. 改进思考

**如果让你重做今日清单，你会怎么改进？**

> 我会增加"智能排序"功能——根据用户历史行为预测今日优先级，比如"经常早上打卡的习惯"自动排在前面。另外可以增加"时间块"视图，让用户按时间段（早晨/下午/晚上）组织任务，更符合实际使用场景。

### 3.3 标签系统

#### 1. 功能定位

**一句话**：通过多标签分类和标签页聚合，让用户按场景（健康/工作/学习）快速聚焦相关习惯，解决"习惯太多找不到"的痛点。

#### 2. 设计决策

| 决策点           | 选项A      | 选项B           | 最终选择    | 决策理由                                           |
| ------------- | -------- | ------------- | ------- | ---------------------------------------------- |
| **标签绑定**      | 单标签（互斥）  | 多标签（交叉）       | **选项B** | 用户调研：一个习惯可能同时属于"健康"和"早晨"，多标签更灵活                |
| **标签页结构**     | 单列表展示    | 双区结构（置顶区+其余区） | **选项B** | 置顶区显示"今日待办"，帮助用户快速聚焦当下任务；其余区显示不应出现/已完成的习惯      |
| **标签页"今日"定义** | 跟随日历选中日期 | 严格指本地今天       | **选项B** | 标签页的核心价值是"快速查看今日某类任务"；如果跟随日历，切换日期后标签页内容变化会造成困惑 |

#### 3. 技术实现

**关键技术方案**：

```typescript
// 标签页双区结构计算（TagsView.vue）
const filteredHabits = computed(() => {
  let habits = habitsStore.habits.filter(h => {
    // 包含未归档且未完成的，以及今日归档的
    if (!h.archived && !h.completed) return true
    if (h.archived && h.archivedAt && isSameLocalDate(h.archivedAt, today)) return true
    return false
  })

  // 按标签筛选
  if (selectedTagId.value) {
    habits = habits.filter(h => h.habitTagIds.includes(selectedTagId.value!))
  }
  
  return habits
})

// 双区划分：置顶区（今日应出现且未完成）+ 其余区
const topZoneItems = computed(() => {
  return filteredHabits.value.filter(h => {
    const shouldShow = shouldAppear(h, today)
    const checkin = todayCheckins.value.get(h.id)
    const isCompleted = checkin && checkin.doneCount >= h.dailyTargetCount
    return shouldShow && !isCompleted
  })
})

const otherZoneItems = computed(() => {
  return filteredHabits.value.filter(h => {
    const shouldShow = shouldAppear(h, today)
    const checkin = todayCheckins.value.get(h.id)
    const isCompleted = checkin && checkin.doneCount >= h.dailyTargetCount
    return !shouldShow || isCompleted  // 不应出现 或 已完成
  })
})
```

**为什么可行**：

- 多标签使用 `habitTagIds: string[]` 数组存储，支持交叉分类
- 双区结构通过 `shouldAppear` 算法动态计算，实时反映今日状态
- 标签页"今日"固定为本地今天，与日历选中日期解耦，避免状态混乱

#### 4. 设计方案演进

| 阶段        | 方案                 | 问题/反馈                          | 演进             |
| --------- | ------------------ | ------------------------------ | -------------- |
| **第一版原型** | 单标签绑定，一个习惯只能打一个标签  | 用户反馈"我的晨跑习惯同时属于健康和早晨，只能选一个不合理" | 改为多标签绑定        |

| **最终版**   | 多标签+双区+固定今日        | 用户反馈"找习惯快多了，置顶区就是我今天要做的"       | 定稿             |

4. **标签删除后习惯怎么处理？** → 从习惯的 `habitTagIds` 中移除该标签ID，习惯本身保留

#### 8. 改进思考

**如果让你重做标签系统，你会怎么改进？**

### 3.4 日历补打卡

| **补打卡范围** | 任意日期可补  | 仅过去日期可补  | **选项B** | 防止用户"预打卡"作弊；未来日期不应提前记录    |
| **补打卡限制** | 所有习惯都可补 | 仅应出现日可补  | **选项B** | 保持数据一致性；"每周一跑步"在周二补打卡不应计入 |

#### 3. 技术实现

**关键技术方案**：

```typescript
// 应出现判定算法（algorithms.ts）
export function shouldAppear(habit: Habit, date: string): boolean {
  // paused状态优先级最高
  if (habit.paused) return false
  
  // 检查基本状态
  if (!habit.enabled || habit.archived || habit.completed) return false
  
  // 检查日期 >= 开始日期
  if (compareDates(date, habit.startDate) < 0) return false
  
  // 频率匹配
  return matchFrequency(habit.frequency.type, date, habit.startDate, habit.frequency)
}

// 补打卡限制（CalendarView.vue）
function canCheckinOnDate(habit: Habit, date: string): { allowed: boolean; reason?: string } {
  // 1. 检查是否是未来日期
  if (isFutureDate(date)) {
    return { allowed: false, reason: '不能补打卡未来日期' }
  }
  
  // 2. 检查是否是应出现日
  if (!shouldAppear(habit, date)) {
    return { 
      allowed: false, 
      reason: `该习惯按${getFrequencyText(habit.frequency)}频率设置，此日期不应出现` 
    }
  }
  
  return { allowed: true }
}
```

**为什么可行**：

- 复用 `shouldAppear` 算法，确保补打卡限制与日常展示逻辑一致
- 限制原因动态生成，根据习惯的频率配置给出具体解释
- 日期使用本地日历天（YYYY-MM-DD），避免时区问题

#### 3. 技术实现

**关键技术方案**：

```typescript
// 统计计算核心（useStats.ts）
export function useStats() {
  // 计算周期统计
  async function calculatePeriodStats(
    period: StatsPeriod,
    date: string
  ): Promise<PeriodStats> {
    const { start, end } = getPeriodRange(period, date)
    
    // 获取周期内的习惯统计
    const habitsStats: HabitStatItem[] = []
    for (const habit of habits) {
      const stats = await calculateHabitStats(habit, start, end, allCheckins, false)
      if (stats.periodTargetCount > 0 || stats.dailyRecords.size > 0) {
        habitsStats.push(stats)
      }
    }
    
    // 计算趋势数据（用于图表）
    const trendData: TrendDataItem[] = []
    const dates = generateDateRange(start, end)
    for (const d of dates) {
      let totalDone = 0
      let totalTarget = 0
      let completedHabitsCount = 0
      
      for (const stat of habitsStats) {
        const record = stat.dailyRecords.get(d)
        if (record) {
          totalDone += record.doneCount
          totalTarget += record.targetCount
          if (record.completed) completedHabitsCount++
        }
      }
      
      trendData.push({
        date: d,
        label: getTrendLabel(d, period),
        totalDone,
        totalTarget,
        completion: totalTarget > 0 ? totalDone / totalTarget : 0,
        completedHabitsCount
      })
    }
    
    return { period, startDate: start, endDate: end, habitsStats, trendData, ... }
  }
  
  // 缓存机制
  const statsCache = new Map<string, PeriodStats>()
  function clearCache() { statsCache.clear() }
}
```

**为什么可行**：

- 统计计算基于 `shouldAppear` 算法，确保与今日清单展示逻辑一致
- 使用缓存机制（statsCache）避免重复计算，提升性能
- 趋势数据预计算，图表直接消费，减少渲染时计算量

## 3.6 方法论沉淀

### 核心设计原则

1. **可解释性优先**：所有限制和规则都必须在产品内可达，用户有权知道"为什么"
2. **损失厌恶保护**：基于心理学原理，保护用户已获得的成就感，降低放弃率
3. **渐进式反馈**：将长期目标拆解为短期可感知的小成就（3天一个阶段）
4. **视图隔离**：不同页面/日期的状态互不干扰，避免用户困惑

### 面试故事套路

1. **STAR结构**：Situation（痛点）→ Task（目标）→ Action（行动/决策）→ Result（成果）
2. **决策对比**：每个关键决策都呈现选项A/选项B/选择理由，体现思考过程
3. **用户反馈**：引用具体用户原话或标注"用户反馈概括"
4. **技术选型**：面试话术聚焦"为什么选这个方案"，而非代码细节

### 可复用的方法论

| 场景   | 方法论         | 应用               |
| ---- | ----------- | ---------------- |
| 激励设计 | 损失厌恶 + 渐进反馈 | 拼豆激励的权重冻结机制      |
| 列表排序 | 视图隔离 + 持久化  | 今日清单和标签页的独立排序    |
| 数据限制 | 可解释性 + 明确提示 | 补打卡的限制提示         |
| 统计口径 | 一致性 + 可追溯   | 复用shouldAppear算法 |

> **Task**：设计统一的"今日"视图，让用户在一个页面管理所有当日任务。
>
> **Action**：
>
> 1. 需求分析发现"混合待办型"用户占比高，既有固定习惯也有临时事项
> 2. 设计"混排+视觉区分"方案：习惯和临时事项在同一列表，后者用虚线边框标识
> 3. 两者同等排序权重，支持拖拽调整；规则排序不覆盖手动排序
> 4. 排序状态按视图key隔离持久化（`today_YYYY-MM-DD`、`tag_<id>_todo`）
>
> **Result**：用户反馈"临时事项和习惯在一起方便多了，不用在多个App间切换"。

## 三、高频追问及标准答案

| 面试官问题                     | 回答要点（30秒内）                                               |
| ------------------------- | -------------------------------------------------------- |
| **为什么是15权重分5阶段？**         | 每阶段3权重≈3天一个小成就，符合**即时反馈原则**；太短没挑战，太长易放弃                  |
| **权重冻结怎么实现的？**            | Checkin记录存储`weightEarned`布尔值，基于**打卡当日目标**判定，与后续目标变更解耦    |
| **5个测试用户怎么选的？**           | 覆盖目标画像：2学生（考研/考公）、2职场人、1自由职业者，完整流程测试                     |
| **为什么用CSS Grid而非Canvas？** | ①DOM节点可控 ②调试方便 ③动画细腻（blur/opacity）；225个节点性能足够            |
| **双区结构怎么计算？**             | 置顶区=应出现且未完成；其余区=不应出现或已完成；复用`shouldAppear`算法              |
| **排序持久化怎么实现？**            | Settings store存储`viewSortOrders`，按视图key隔离，支持今日清单和标签页独立排序 |
| **自定义频率怎么计算？**            | `(date - startDate)天数差 % N == 0`，从起始日期起算每N天出现一次          |
| **统计缓存怎么更新？**             | 监听`checkin:completed`事件，自动清除`statsCache`并刷新              |

## 四、设计原则速记（4句话）

1. **可解释性优先** → 所有限制必须在产品内可查，用户有权知道"为什么"
2. **损失厌恶保护** → 保护已得成就感，权重冻结降低断签后放弃率
3. **渐进式反馈** → 长期目标拆解为3天小成就，持续给予正向激励
4. **视图隔离** → 不同页面/日期状态互不干扰，避免用户困惑
