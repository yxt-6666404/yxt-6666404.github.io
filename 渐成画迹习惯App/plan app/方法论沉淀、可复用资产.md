## 4.7 AI PM核心能力模型

### 从6个真实问题中提炼的3个核心能力

在开发过程中遇到的6个问题（自适应布局、提醒样式、日历提醒、长按弹窗位置、长按弹窗滑动、移动端提醒），让我提炼出AI PM必备的3个核心能力：

#### 能力一：迭代能力——从临时修复到系统规范

**问题来源**：开发过程中发现的多类问题（自适应布局、弹窗定位、移动端限制等）

**能力定义**：不满足于"修完就行"，而是发现问题模式→沉淀系统规范→建立预防机制的完整闭环能力

**关键动作**：

1. **开发中（点状修复）**：遇到问题时提醒AI临时修复（如提醒自适应、修复弹窗位置）
2. **开发后（模式识别）**：发现这些不是孤立问题，而是缺乏系统性约束
3. **规范沉淀**：协同AI编写《响应式自适应布局实现指南》，统一解决所有类似问题
4. **流程嵌入**：将规范融入验收清单和后续Prompt，预防复现

**真实案例**：

开发中发现激励页面小字换行、设置页面文字截断、统计页面重叠等问题。当时只是零散提醒AI"注意自适应"，但没有发现的页面就没有处理。开发完成后，我意识到这是系统性问题——**不是"修了几个页面"，而是"缺乏全局约束"**。于是我协同AI说明之前所有注意的问题，沉淀了《响应式自适应布局实现指南》，让AI在不影响功能的情况下统一调整所有页面。

**面试话术**：

> "开发中我发现激励页面小字换行、设置页面截断等问题，当时只是零散提醒AI修复。但开发完成后我意识到：这不是孤立问题，而是缺乏系统性约束。于是我沉淀了《响应式自适应布局实现指南》，把'禁止1-3字换行''必须用Tailwind响应式类'等约束规范化，让AI统一调整所有页面。这不是'修了几个Bug'，而是'建立了预防机制'。"

#### **能力二：AI协作能力——"国际版+中文版"双循环+模型选择策略**

遇到三大坑：①token中断导致上下文丢失，恢复后文档膨胀；②单文档膨胀混淆边界，PRD里混进技术设计、数据库定义；③AI对业务规则理解弱（比如"完成趋势"提示需要举例子+图片说明才能理解）；④中文版AI生成提醒样式简陋，修改两次效果不好。

**沉淀机制一：模型选择策略**

| 任务类型 | 推荐模型  | 原因             |
| :--- | :---- | :------------- |
| 视觉设计 | 国际版AI | 训练数据包含更多现代UI案例 |
| 功能逻辑 | 均可    | 两个模型都能胜任       |
| 原型理解 | 国际版AI | 对HTML结构理解更准确   |
| 代码生成 | 均可    | 根据具体语言选择       |

具体案例：中文版AI生成提醒样式很丑，保持Prompt不变切换至国际版AI一次就生成美观设计。这让我形成策略：视觉设计用国际版，功能逻辑用中文版，模型选择很重要。

**沉淀机制二：四分文档交叉验证**

"国际版发散覆盖→拆分四分文档→中文版补PRD核心规则→skill交叉验证→Kimi最终检查"。国际版负责发散，中文版负责收敛精确，四分文档相互引用补充，Kimi检查完整性。解决文档一致性问题，确保需求层完整、各层引用一致。

**沉淀机制三：双线开发策略**

国内版和国际版分工明确：国内版负责做页面和基本逻辑，国际版负责修复难题（国内版3-4次解决不了的问题后切换）。这种任务分流策略充分利用两个模型的能力边界，国内版擅长中文理解和页面生成，国际版擅长视觉设计和复杂问题修复。

**沉淀机制四：详细Prompt工程**

开发时的结构化Prompt包含7部分：①项目路径（habit-tracker目录结构）；②参考文档（PRD/TDD/DBD/TestPlan引用）；③原型路径（HTML原型文件路径，显式引用）；④验收范围（本次task对应哪些AC）；⑤Skills组合（using-superpowers + request-analyzer + frontend-design + vue-best-practices + vue-pinia-best-practices + create-adaptable-composable + vue-testing-best-practices + test-driven-development + Systematic Debugging）；⑥开发流程（先读文档→再改代码→最后写测试）；⑦技术约束（Tailwind断点、禁止1-3字换行、触摸目标44px等）。

**能力跃迁**：从"只会问帮我写个XX功能"到"建立工作流（文档→原型→开发→验收）"，从"单模型使用"到"多模型协作+双线开发"。

**面试话术**：

**【对应问题】"你和AI怎么协作？" / "AI辅助开发有什么坑？"**

> "我和AI协作遇到三个坑：token中断导致上下文丢失、单文档膨胀混淆边界、AI对业务规则理解弱。我还发现中文版生成提醒样式简陋，修改两次效果不好，保持Prompt不变切换至国际版AI一次就生成美观设计。这让我沉淀了两个机制：一是模型选择策略——视觉设计用国际版、功能逻辑用中文版；二是四分文档交叉验证——国际版发散→中文版补PRD→skill验证→Kimi检查，解决需求层与实现层混淆问题。"

#### **能力三：需求管理能力——"原型驱动的需求迭代"+PRD完整性意识**

用"原型先行"策略和"AI交叉验证+人工走查"方法验证需求。三层协作工具链：①Trae用`frontend-design`/`theme-factory`生成设计系统；②输出HTML原型，用Figma的html-to-design插件导入做整体修改；③局部修改用Pencil MCP画完截图回Trae迭代。文档修改同步回写PRD，Trae和Kimi双边验证。

**关键发现**：

1. AI生成的原型缺少"空状态"和"加载态"，主动补充**补齐关键细节**
2. **UI设计倒逼需求细化**：统计系统PRD最初只写"提供日/周/月/年统计"，但画原型时发现不够——需要周期选择器、多维度视图、趋势图、习惯榜单、日期选择功能。这些交互细节在PRD里完全没定义，如果不画原型，开发时才会发现缺需求。
3. **日历提醒逻辑缺失教训**：日历提醒在原型里有交互设计，但PRD只写了"支持提醒"1句话，没定义具体交互。AI开发时只读了PRD，没实现提醒功能。这是"AI不会读原型"的典型问题。

**沉淀机制一：PRD完整性原则**

1. **AI读原型≠读所有内容**：AI可能遗漏原型中的细节
2. **关键需求必须写入PRD**：不能依赖AI"理解"原型

作为独立开发者，我没有团队协作的分支规范压力，所以采用了适合自己节奏的三分支简化工作流：

- **main分支**：承载核心产出，四分文档（PRD/TDD/DBD/TestPlan）和P0核心代码都在这里，确保主分支始终是项目的"完整状态"
- **dev分支**：用于bug修复和POC尝试（如PWA功能验证），快速试错不影响主线
- **release分支**：专门用于发布部署到Vercel production域名，对外正式访问

main和dev分支的每次推送都会在Vercel的预览域名里面展示实际效果，方便我快速验证修改。只有确认稳定后才合并到release分支上生产。

这种简化工作流的好处是：

1. **快速POC验证**：比如PWA功能在dev分支尝试，发现iOS/鸿蒙兼容性不稳定后直接在dev分支放弃，不污染main
2. **预览验证→生产发布**：每次修改先在预览域名看到效果，确认没问题才上生产
3. **适合自己节奏**：没有团队代码审查、分支合并审批流程，独立开发可以更灵活

> 独立开发不等于"不规范"。三分支工作流让我在灵活性和安全性之间找到平衡——main直接开发，dev做POC和修复，release部署生产。Vercel预览域名自动验证每次推送，相当于CI/CD。团队开发我会用feature分支+PR review，但独立场景下这个流程更高效。

## 5.4 个人成长与能力沉淀

#### 从"完美主义"到"MVP思维"的思维转变

**项目前的思维陷阱**：

- 总想把所有功能都做完美，觉得"少了就不像产品"
- 遇到功能犹豫时，默认选择"加上总比没有好"
- 容易被"竞品有这个功能"说服自己做额外功能

**MVP思维的建立**：

通过P0范围划定+开发过程的持续克制，我学会了两个MVP思维的核心原则：

1. **先验证核心价值，再考虑锦上添花**
   - **功能层**：云同步虽然价值高但成本高，通过模型确定放p1，在P0先验证核心价值“同步“—使用IndexedDB本地，通过用户去实现同步，满足用户更关心"数据会不会丢"而不是"能不能导出"，同Repository模式已预留扩展，P1可低成本接入Supabase；
   - **开发层**：不提前做过度设计——统计系统先做日/周/月基础维度，不急于加多维度的趋势图/对比图；标签系统先做分类管理，不做颜色自定义；拼豆激励用CSS clip-path简单切换，不做复杂动画过渡；**结果：1.5周完成开发+测试，而不是陷入细节打磨2周**
   - **原则**：延迟满足高成本功能+开发克制，先证明产品有人用，再逐步完善
2. **识别"反向需求"，敢于做减法**
   - 案例：社交/排行榜明确排除——竞品Forest用户反馈"太关注别人反而更焦虑"，与"零压力自我对话"定位冲突，做了反而降低满意度
   - 原则：不是"功能越多越好"，而是"功能越准越好"

**面试话术**：

> "这个项目让我从'完美主义'转变为'MVP思维'。以前总想把所有功能都做完美，现在学会了2件事：一是先验证核心价值再考虑锦上添花，比如云同步虽然好但成本高先验证产品有人用再做，开发时也克制——统计系统先做基础维度不加趋势图、标签系统先做分类不做颜色自定义、拼豆用简单切换不做复杂动画；二是识别反向需求敢于做减法，社交/排行榜做了反而增加焦虑。MVP思维的核心不是'做得少'，而是'做得准'。"

### 面试话术

**【对应问题】"你从这个项目中学到了什么？" / "你说的'独立做产品'具体体现在哪？" / "你最大的成长是什么？"**

**【精简版】（30秒）**：

> 这个项目让我从"能写代码"成长为"能独立做产品"。最大的转变是从完美主义到MVP思维——学会先验证核心价值再考虑锦上添花，识别反向需求敢于做减法。同时沉淀了3个AI PM核心能力（迭代能力、AI协作能力、需求管理能力）和6项方法论与规范（4项技术规范+2项核心机制）。

**【故事版】（100字）**：

> **Situation**：项目前只注重代码开发，有完美主义倾向，总觉得"少了就不像产品"。
>
> **Task**：独立完成从需求到交付的全流程，学会在有限时间内做正确决策。
>
> **Action**：经历P0/P1范围划定，建立模型选择策略和四分文档交叉验证机制；用原型先行策略倒逼需求细化；经历需求分析、产品决策、质量保障、用户验证全流程，沉淀3个核心能力和6项方法论与规范。
>
> **Result**：从执行者成长为能独立做产品的AI PM。

**【追问展开版】（200字）**：

> **MVP思维转变**：以前我总想把所有功能都做完美，遇到功能犹豫时默认选择"加上总比没有好"。但通过P0范围划定+开发过程中的持续克制，我学会了两个维度的MVP思维：
>
> **产品维度**：一是先验证核心价值，云同步虽然好但成本高，选择P0先验证产品有人用，使用IndexedDB本地，通过用户去实现同步Repository模式已预留扩展；二是识别反向需求，社交/排行榜竞品Forest用户反馈"太关注别人反而更焦虑"，明确排除。
>
> **开发维度**：不提前做过度设计——统计系统先做日/周/月基础维度，不急于加趋势图/对比图；标签系统先做分类管理，不做颜色自定义；拼豆激励用CSS clip-path简单切换，不做复杂动画过渡。这些"克制"让我1.5周完成开发+测试，而不是陷入细节打磨。
>
> **核心机制一：模型选择策略+四分文档交叉验证**
>
> 遇到中文版AI生成提醒样式简陋，切换国际版一次就好，沉淀"视觉设计用国际版，功能逻辑用中文版"策略。国际版AI单文档膨胀后，拆分为四分文档但第一次拆分发现PRD缺失，于是用中文版补充核心规则，skill建立引用，Kimi检查。沉淀机制："国际版发散→中文版收敛→四分交叉验证→Kimi检查"。
>
> **核心机制二：原型驱动需求迭代+PRD完整性**
>
> 用"原型先行"策略，三层协作工具链（Trae生成设计→Figma整体修改→Pencil局部迭代）。关键发现：统计系统画原型时发现需要周期选择器、多维度视图等7项交互，PRD里完全没定义；日历提醒原型有但PRD没写，AI就没实现。建立机制：功能修改同步回写PRD，关键需求必须写入文档。
>
> **能力跃迁**：从"被动接收需求"到"主动发现痛点"，从"关注技术实现"到"用Kano模型+2×2矩阵做优先级决策"，从"开发完手动点点"到"三层测试金字塔+验收驱动开发"。
>
> **下次怎么做**：先做减法再做加法（列"可以不做"清单）、开发克制建立检查点（每完成一个功能问自己"这是P0必须的吗"）、遇到第2个同类问题时沉淀规范而非只修复。

#### 预埋追问点

| 面试官问题                    | 回答要点                                      |
| ------------------------ | ----------------------------------------- |
| **你说的"独立做产品"具体体现在哪？**    | 能独立完成需求洞察→产品决策→开发实现→质量保障→用户验证全流程，而不是只负责编码 |
| **3个能力哪个最重要？**           | 迭代能力最重要——发现问题后沉淀规范而非重复修复，从临时修复到系统预防机制     |
| **方法论真的能复用到其他项目吗？**      | 能，四分文档工作流和Prompt模板已在后续小项目中验证有效            |
| **你觉得自己还缺什么能力？**         | 商业化思维和数据分析能力——目前更关注产品功能，对用户增长和变现模式思考还不够深入 |
| **如果让你给AI PM新手建议，会说什么？** | "先建立文档习惯，再学Prompt。文档是思考的载体，Prompt只是工具。"   |
