### 4.2.1 数据库索引优化：查询性能提升10倍

#### 发现过程

**Day 5**：在测试"日历视图"加载30天历史数据时，发现页面有明显的加载延迟（\~2秒）。通过Chrome DevTools Performance面板定位到数据库查询瓶颈——查询某习惯某月的打卡记录时，Dexie.js进行了全表扫描。

#### 根因分析

```typescript
// ❌ 问题：没有索引，全表扫描
// 查询某习惯某月的打卡记录需要遍历整个checkins表
const checkins = await db.checkins
  .where('habitId')
  .equals(habitId)
  .and(checkin => checkin.date.startsWith('2024-03'))  // 全表扫描！
  .toArray()
```

#### 修复方案

```typescript
// ✅ 添加复合索引
// db/index.ts
this.version(1).stores({
  checkins: '++id, [habitId+date], date',  // 复合索引
  habits: '++id, archived, completed',
  // ...
})

// 查询性能提升10倍
const checkins = await db.checkins
  .where('[habitId+date]')  // 使用复合索引
  .between([habitId, '2024-03-01'], [habitId, '2024-03-31'])
  .toArray()
```

#### 验证结果

- **优化前**：30天数据查询\~2000ms
- **优化后**：30天数据查询\~180ms（提升11倍）

### 4.2.2 权重冻结：防止降低目标倒灌

#### 发现过程

**Day 7 设计评审**：在评审激励系统设计方案时，发现一个逻辑漏洞：

1. 用户设置目标8次/天，完成8次打卡，获得1权重
2. 用户修改目标为2次/天
3. 问题：已获得的权重是否应该保留？

> 注：这是设计评审时发现的产品逻辑问题，不是开发后的技术Bug。

#### 决策分析

| 选项         | 逻辑                                  | 问题                |
| ---------- | ----------------------------------- | ----------------- |
| **A：权重回退** | 按新目标重新计算，8次/2次=4权重，但每天最多1权重，所以保留1权重 | 用户感到"被剥夺"，损失厌恶心理  |
| **B：权重冻结** | 已获得的权重永不回退                          | 保护用户成就感，符合损失厌恶心理学 |

**最终选择：B（权重冻结）**

#### 实现方案

```typescript
// algorithms.ts
export function canEarnWeight(
  doneCount: number,
  targetCount: number,
  alreadyEarned: boolean
): boolean {
  if (alreadyEarned) return false  // 已获得过，不再重复获得
  if (targetCount <= 0) return false
  return doneCount >= targetCount   // 基于打卡当日目标判定
}

// Checkin记录存储weightEarned布尔值
interface Checkin {
  // ...
  weightEarned: boolean  // 一旦获得，永不回退
}
```

#### 验证结果

- **单元测试**：权重冻结逻辑通过所有测试用例
- **用户反馈**：测试用户表示"这样很合理，不会因为改目标而觉得亏了"

### 4.2.3 自适应布局多次调整：从反复修复到方法论沉淀

#### 发现过程

**Day 4-6**（开发第4-6天）：

> 在验收页面时发现多处布局问题：激励页面的小字提示出现"每"字单独一行，设置页面的提醒功能说明出现"每日目"三字换行的尴尬排版。AI开发中多次忘记自适应要求，导致移动端和桌面端的字体大小没有区分。

**具体问题案例**：

- 激励页面："此刻即永恒 — 及时保存才能永久留存，进入下一阶段后将不可复现"
- 设置页面："提醒功能说明"请确保已在系统设置中开启相关通知权限”"
- 统计页面：小屏手机上文字被截断或重叠

#### 根因分析

**问题根源**：

1. **AI遗忘约束**：AI在生成代码时，虽然Prompt中写了"响应式自适应"，但具体实现时经常忘记
2. **px写死问题**：多处使用`px`单位写死字体大小，没有使用Tailwind响应式类
3. **缺乏换行约束**：没有强制要求"绝对不能出现1-3字换行"

#### 修复方案

**第一步：问题收集与分类**
收集所有出现布局问题的页面和组件，分类整理：

- 字体大小未响应式调整
- 文本换行不合理
- 触摸目标大小不足44px

**第二步：研究Tailwind响应式策略**

```javascript
// tailwind.config.js 断点配置
screens: {
  sm: '640px',   // 大屏手机
  md: '768px',   // 平板/小桌面 ← 主要断点
  lg: '1024px',  // 桌面
  xl: '1280px',  // 大桌面
}
```

**第三步：编写《响应式自适应布局实现指南》**
沉淀规范文档，强制约束：

1. **字体大小自适应**：移动端`text-sm`，桌面端`md:text-base`
2. **绝对禁止1-3字换行**：使用`whitespace-nowrap`或调整文案
3. **必须用Tailwind响应式类**：禁止用`px`写死
4. **触摸目标至少44px**：符合Apple HIG和Material Design要求

**关键技巧代码示例**：

```html
<!-- 字体大小响应式 -->
<p class="text-sm md:text-base">习惯描述</p>

<!-- 强制不换行（短文本） -->
<span class="whitespace-nowrap text-xs md:text-sm">每日目标</span>

<!-- Flex子元素必须加min-w-0防止溢出 -->
<div class="flex">
  <div class="flex-1 min-w-0 truncate">很长的文字...</div>
</div>
```

**Prompt优化经验**：

```
"XXXX地方的移动端和桌面端自适应布局，改变字体大小，
两个端口字体大小不一样，绝对不能出现1-3字换行，
位置不够要在语句合理处换行，不能用px写死，
要有Tailwind响应式类"
```

#### 验证结果

- **问题修复率**：100%（所有发现的问题都已修复）
- **规范沉淀**：《响应式自适应布局实现指南》成为后续开发标准
- **预防效果**：后续开发不再出现类似换行问题

### 4.2.6 长按弹窗位置问题：交互细节的精准定位

#### 发现过程

**Day 6**（开发第6天）：

> 在测试今日清单的长按交互时，发现长按习惯卡片后弹出的操作菜单（编辑/删除/归档）位置不对——它出现在屏幕左上角，而不是长按点的右下方。这导致用户需要移动手指才能点击菜单项，体验很差。

#### 根因分析

**问题根源**：

1. **定位计算错误**：使用`event.clientX/clientY`获取的是视口坐标，没有考虑滚动偏移
2. **缺少边界检测**：没有处理弹窗超出屏幕边界的情况
3. **坐标系混淆**：鼠标/触摸事件的坐标系与CSS定位的坐标系未正确转换

#### 修复方案

```typescript
// 修复后的定位逻辑
function calculatePopupPosition(
  touchX: number,
  touchY: number,
  popupWidth: number,
  popupHeight: number
): { x: number; y: number } {
  // 考虑滚动偏移，获取相对于文档的坐标
  const scrollX = window.scrollX || window.pageXOffset
  const scrollY = window.scrollY || window.pageYOffset
  
  let x = touchX + scrollX + 10 // 右偏移10px
  let y = touchY + scrollY + 10 // 下偏移10px
  
  // 边界检测：防止超出屏幕右边界
  if (x + popupWidth > window.innerWidth + scrollX) {
    x = touchX + scrollX - popupWidth - 10 // 改为左偏移
  }
  
  // 边界检测：防止超出屏幕下边界
  if (y + popupHeight > window.innerHeight + scrollY) {
    y = touchY + scrollY - popupHeight - 10 // 改为上偏移
  }
  
  return { x, y }
}
```

#### 验证结果

- **位置准确性**：弹窗始终出现在长按点的右下方（或自动调整方向避免溢出）
- **边界处理**：在屏幕边缘长按也能正确显示弹窗
- **用户体验**：用户手指无需移动即可点击菜单项

### 4.2.7 长按弹窗随页面滑动问题：优雅的降级方案

#### 发现过程

**Day 6**（开发第6天）：

> 修复位置问题后，又发现新问题：当弹窗显示时，如果用户滚动页面，弹窗会跟着页面一起移动，而不是固定在长按点位置。这看起来像是弹窗"粘"在屏幕上，视觉效果很奇怪。

#### 根因分析

**问题根源**：

1. **定位方式选择**：使用`position: absolute`相对于文档定位，页面滚动时自然跟随
2. **固定定位的代价**：改用`position: fixed`虽然可以解决跟随问题，但需要重新计算相对于视口的位置
3. **复杂场景处理**：页面滚动后，原长按点可能已经不在视口内，弹窗显示逻辑变得复杂

#### 修复方案

**方案对比**：

| 方案             | 实现复杂度      | 用户体验      | 最终选择 |
| -------------- | ---------- | --------- | ---- |
| A. 改为`fixed`定位 | 高（需重新计算坐标） | 好         | ❌    |
| B. 滚动时自动关闭弹窗   | 低          | 好（符合用户预期） | ✅    |
| C. 实时更新弹窗位置    | 中（需监听滚动事件） | 一般        | ❌    |

**选择方案B的原因**：

- 用户滚动页面通常意味着"我不想操作这个弹窗了"
- 自动关闭符合用户心理预期
- 实现简单，无需复杂的坐标计算

**实现代码**：

```typescript
// 监听滚动事件，自动关闭弹窗
useEffect(() => {
  const handleScroll = () => {
    if (isPopupVisible) {
      closePopup() // 滚动时自动关闭
    }
  }
  
  window.addEventListener('scroll', handleScroll)
  return () => window.removeEventListener('scroll', handleScroll)
}, [isPopupVisible])
```

#### 验证结果

- **实现简洁性**：代码简单，无需复杂坐标计算
- **用户反馈**：测试用户表示"这样很自然，滚动就是想取消"

### 4.2.8 移动端提醒不可用问题：平台限制与降级方案

#### 发现过程

**Day 9**（开发第9天）：

> 在真机测试提醒功能时，发现设置的提醒时间到了却没有触发。排查后发现：移动端浏览器在后台会冻结JavaScript执行，定时任务（`setTimeout`/`setInterval`）无法正常运行。这是浏览器的省电策略，无法绕过。

#### 根因分析

**平台限制的本质**：

1. **iOS Safari**：后台标签页JavaScript被暂停，定时器冻结
2. **Android Chrome**：后台页面CPU限制，定时器延迟执行或冻结
3. **PWA限制**：即使添加到主屏幕，后台行为与普通网页一致
4. **技术约束**：Web标准没有提供后台持续运行的能力

**影响范围**：

- 定时提醒功能在移动端浏览器无法可靠运行
- 只有App（原生或Capacitor/Ionic）才能实现后台提醒

#### 修复方案

**方案调研与选择**：

| 方案                      | 可行性 | 工作量 | 兼容性           | 最终选择 |
| ----------------------- | --- | --- | ------------- | ---- |
| A. PWA + Service Worker | 低   | 大   | iOS/鸿蒙不支持后台同步 | ❌    |
| B. Web Push API         | 中   | 大   | 需要服务器支持       | ❌    |
| C. 文档说明+预留Capacitor接口   | 高   | 小   | 打包后全平台支持      | ✅    |

**PWA方案评估**（参考 `移除PWA.md`）：

- 调研发现PWA在iOS和鸿蒙系统上后台同步兼容性不稳定
- MVP阶段优先保证核心功能体验，避免过度工程化
- 采用"框架效应"话术：不说"没做PWA"，而说"优先做核心功能"

## 4.7 AI PM核心能力模型

#### 能力一：迭代能力——从临时修复到系统规范

**问题来源**：开发过程中发现的多类问题（自适应布局、弹窗定位、移动端限制等）

**能力定义**：不满足于"修完就行"，而是发现问题模式→沉淀系统规范→建立预防机制的完整闭环能力

**关键动作**：

1. **开发中（点状修复）**：遇到问题时提醒AI临时修复（如提醒自适应、修复弹窗位置）
2. **开发后（模式识别）**：发现这些不是孤立问题，而是缺乏系统性约束
3. **规范沉淀**：协同AI编写《响应式自适应布局实现指南》，统一解决所有类似问题
4. **流程嵌入**：将规范融入验收清单和后续Prompt，预防复现

**真实案例**：

开发中发现激励页面小字换行、设置页面文字截断、统计页面重叠等问题。当时只是零散提醒AI"注意自适应"，但没有发现的页面就没有处理。开发完成后，我意识到这是系统性问题——**不是"修了几个页面"，而是"缺乏全局约束"**。于是我协同AI说明之前所有注意的问题，沉淀了《响应式自适应布局实现指南》，让AI在不影响功能的情况下统一调整所有页面。

**面试话术**：

> "开发中我发现激励页面小字换行、设置页面截断等问题，当时只是零散提醒AI修复。但开发完成后我意识到：这不是孤立问题，而是缺乏系统性约束。于是我沉淀了《响应式自适应布局实现指南》，把'禁止1-3字换行''必须用Tailwind响应式类'等约束规范化，让AI统一调整所有页面。这不是'修了几个Bug'，而是'建立了预防机制'。"

#### **能力二：AI协作能力——"国际版+中文版"双循环+模型选择策略**

遇到三大坑：①token中断导致上下文丢失，恢复后文档膨胀；②单文档膨胀混淆边界，PRD里混进技术设计、数据库定义；③AI对业务规则理解弱（比如"完成趋势"提示需要举例子+图片说明才能理解）；④中文版AI生成提醒样式简陋，修改两次效果不好。

**沉淀机制一：模型选择策略**

| 任务类型 | 推荐模型  | 原因             |
| :--- | :---- | :------------- |
| 视觉设计 | 国际版AI | 训练数据包含更多现代UI案例 |
| 功能逻辑 | 均可    | 两个模型都能胜任       |
| 原型理解 | 国际版AI | 对HTML结构理解更准确   |
| 代码生成 | 均可    | 根据具体语言选择       |

具体案例：中文版AI生成提醒样式很丑，保持Prompt不变切换至国际版AI一次就生成美观设计。这让我形成策略：视觉设计用国际版，功能逻辑用中文版，模型选择很重要。

**沉淀机制二：四分文档交叉验证**

"国际版发散覆盖→拆分四分文档→中文版补PRD核心规则→skill交叉验证→Kimi最终检查"。国际版负责发散，中文版负责收敛精确，四分文档相互引用补充，Kimi检查完整性。解决文档一致性问题，确保需求层完整、各层引用一致。

**沉淀机制三：双线开发策略**

国内版和国际版分工明确：国内版负责做页面和基本逻辑，国际版负责修复难题（国内版3-4次解决不了的问题后切换）。这种任务分流策略充分利用两个模型的能力边界，国内版擅长中文理解和页面生成，国际版擅长视觉设计和复杂问题修复。

**沉淀机制四：详细Prompt工程**

开发时的结构化Prompt包含7部分：①项目路径（habit-tracker目录结构）；②参考文档（PRD/TDD/DBD/TestPlan引用）；③原型路径（HTML原型文件路径，显式引用）；④验收范围（本次task对应哪些AC）；⑤Skills组合（using-superpowers + request-analyzer + frontend-design + vue-best-practices + vue-pinia-best-practices + create-adaptable-composable + vue-testing-best-practices + test-driven-development + Systematic Debugging）；⑥开发流程（先读文档→再改代码→最后写测试）；⑦技术约束（Tailwind断点、禁止1-3字换行、触摸目标44px等）。

**能力跃迁**：从"只会问帮我写个XX功能"到"建立工作流（文档→原型→开发→验收）"，从"单模型使用"到"多模型协作+双线开发"。

**面试话术**：

**【对应问题】"你和AI怎么协作？" / "AI辅助开发有什么坑？"**

> "我和AI协作遇到三个坑：token中断导致上下文丢失、单文档膨胀混淆边界、AI对业务规则理解弱。我还发现中文版生成提醒样式简陋，修改两次效果不好，保持Prompt不变切换至国际版AI一次就生成美观设计。这让我沉淀了两个机制：一是模型选择策略——视觉设计用国际版、功能逻辑用中文版；二是四分文档交叉验证——国际版发散→中文版补PRD→skill验证→Kimi检查，解决需求层与实现层混淆问题。"

#### **能力三：需求管理能力——"原型驱动的需求迭代"+PRD完整性意识**

用"原型先行"策略和"AI交叉验证+人工走查"方法验证需求。三层协作工具链：①Trae用`frontend-design`/`theme-factory`生成设计系统；②输出HTML原型，用Figma的html-to-design插件导入做整体修改；③局部修改用Pencil MCP画完截图回Trae迭代。文档修改同步回写PRD，Trae和Kimi双边验证。

**关键发现**：

1. AI生成的原型缺少"空状态"和"加载态"，主动补充**补齐关键细节**
2. **UI设计倒逼需求细化**：统计系统PRD最初只写"提供日/周/月/年统计"，但画原型时发现不够——需要周期选择器、多维度视图、趋势图、习惯榜单、日期选择功能。这些交互细节在PRD里完全没定义，如果不画原型，开发时才会发现缺需求。
3. **日历提醒逻辑缺失教训**：日历提醒在原型里有交互设计，但PRD只写了"支持提醒"1句话，没定义具体交互。AI开发时只读了PRD，没实现提醒功能。这是"AI不会读原型"的典型问题。

**沉淀机制一：PRD完整性原则**

1. **AI读原型≠读所有内容**：AI可能遗漏原型中的细节
2. **关键需求必须写入PRD**：不能依赖AI"理解"原型
3. **原型→PRD→开发**：建立完整的流程规范

**沉淀机制二：UI产出同步回写**

**UI产出过程中，功能修改同步回写PRD**。用`request-analyzer`分析HTML原型功能，用`doc-coauthoring`更新PRD对应章节，3天完成35个原型+PRD同步更新。

**能力跃迁**：从"我觉得好用就行"到"5位用户验证，数据驱动迭代"，从"原型驱动"到"文档化规范"。

**面试话术**：

**【对应问题】"你怎么保证需求不被改来改去？" / "原型和PRD的关系是什么？"**

> "我用原型先行策略验证需求，三层协作工具链：Trae生成设计→Figma整体修改→Pencil局部迭代。关键发现是UI设计倒逼需求细化——统计系统PRD最初只写'提供日周月年统计'，但画原型时发现需要周期选择器、多维度视图等7项交互。还有个教训：日历提醒原型有但PRD没写，AI就没实现。我建立机制：功能修改同步回写PRD，关键需求必须明确写入文档，不能依赖AI理解原型。"
